<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Google OAuth → app | OAuth callback is the perimeter — NextAuth validates state + PKCE automatically |
| Browser cookie ↔ NextAuth | Session JWT in httpOnly cookie — NEVER accessible to client JS |
| signIn callback → DB | Whitelist check happens server-side in Node runtime with Drizzle query — cannot be bypassed by client |
| Middleware → route | Middleware runs on Edge before any page render — final gate before Server Components see the request |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Spoofing | OAuth callback | mitigate | Auth.js v5 enforces state + PKCE + nonce by default for Google provider; no custom code needed but explicit `pages: { signIn: '/login' }` and no `error` page override preserves built-in error handling |
| T-04-02-02 | Tampering | Session JWT | mitigate | `AUTH_SECRET` (≥256-bit, generated via `openssl rand -base64 32`) signs JWT; cookie is httpOnly + sameSite=lax by NextAuth default; rotating AUTH_SECRET invalidates all sessions |
| T-04-02-03 | Repudiation | Whitelist check audit | accept | Phase 4 does not log whitelist denials to audit_log; Phase 6 owns audit log. Acceptable: denial leaves the user on /acesso-negado, no security event missed for this internal MVP |
| T-04-02-04 | Information disclosure | Email in /acesso-negado URL | accept | UI-SPEC explicitly accepts this tradeoff — internal tool, no PII concern, makes the denied state self-explanatory. Documented as a knowing tradeoff |
| T-04-02-05 | Denial of service | DB lookup on every sign-in | accept | `usuariosPermitidos.email` has a unique index (Phase 3 schema); single indexed lookup per OAuth — negligible cost, no rate-limiting needed at MVP scale |
| T-04-02-06 | Elevation of privilege | Bypassing whitelist | mitigate | Whitelist check is in `signIn` callback (server-only, runs in Node runtime before session is minted). Returning a string URL from the callback FORCES Auth.js to abort session creation and redirect — there is no path where a non-whitelisted email gets a valid session. Additional belt-and-suspenders: `jwt` callback queries the same table again on first sign-in, so even if signIn returned `true` by mistake, role/nome population would fail visibly |
| T-04-02-07 | Information disclosure | Open redirect via signIn callback return | mitigate | Returned redirect URL is a HARDCODED relative path `/acesso-negado?email=...` — no user-controlled redirect target. The `email` query string is encoded via `encodeURIComponent` to prevent it from being interpreted as a different path |
| T-04-02-SC | Tampering | npm install (no new packages) | accept | This plan installs NO new packages — uses already-installed `next-auth@5.0.0-beta.25`, `@neondatabase/serverless`, `drizzle-orm`. No legitimacy gate needed |
</threat_model>
