<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL search params → React rendering | `email` from `?email=` is user-controlled (anyone can craft the URL); React's default escaping prevents XSS |
| Browser → SignOutButton | signOut() destroys the session cookie server-side via NextAuth's signout endpoint |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Information disclosure | Email in URL | accept | UI-SPEC explicitly documents this as a knowing tradeoff for an internal tool. Browser history and HTTP referrer headers will see the email — acceptable because (a) the email is the user's own, (b) tool is internal/private, (c) makes the denial self-explanatory without server-side state |
| T-04-04-02 | Tampering | XSS via `?email=<script>` | mitigate | React auto-escapes `{email}` text content. No `dangerouslySetInnerHTML`. No URL parsing that could mistake the email for a different path |
| T-04-04-03 | Spoofing | A whitelisted attacker craft­ing `?email=victim@…` to confuse social engineering | accept | Reaching /acesso-negado requires no auth; anyone can pass any email in the URL. But the page does no privileged action — it only displays text and offers signOut. No spoofing impact beyond cosmetic confusion |
| T-04-04-04 | Information disclosure | The mailto: link reveals admin email | accept | Admin email `marcelo.mattioli@pruma.io` is intentionally publicly displayed as the contact for access requests. Documented decision (D-11) |
| T-04-04-05 | Denial of service | Page does no DB query | accept | Page is fully static once rendered — no DB load, no rate-limiting concern |
| T-04-04-SC | Tampering | No new package installs in this plan | accept | Uses already-installed `next-auth@5.0.0-beta.25` and `lucide-react`. No legitimacy gate triggered |
</threat_model>
