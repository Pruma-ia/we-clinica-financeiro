<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase | OAuth redirect flow; Supabase validates Google tokens before issuing session |
| Supabase → PostgreSQL | RLS via `is_permitido()` enforces row-level access; anon key cannot bypass RLS |
| Vercel CDN → Browser | Serves static SPA bundle; anon key is embedded in bundle |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | Supabase anon key in Vite bundle | accept | Anon key is public-safe by design; RLS enforces all row-level access. No sensitive data accessible without authenticated whitelist membership. |
| T-01-02 | Spoofing | Google OAuth token forgery | accept | Supabase GoTrue validates Google id_token server-side; client never handles raw OAuth code. |
| T-01-03 | Elevation of Privilege | Whitelist bypass | mitigate | `fetchPerfilPermitido` in `src/lib/auth.js` checks `usuarios_permitidos.ativo=true` before app renders. RLS `is_permitido()` additionally enforces at DB layer. Dual gate: app layer + DB layer. |
| T-01-04 | Tampering | OAuth redirect URI mismatch | mitigate | Exact redirect URI `https://<project-id>.supabase.co/auth/v1/callback` registered in Google Cloud Console. Supabase rejects requests with unregistered redirect URIs. |
| T-01-05 | Information Disclosure | Env vars in Vercel | accept | SUPABASE_URL and SUPABASE_ANON_KEY are public-safe. DB password never exposed in app bundle. |
| T-01-06 | Denial of Service | Supabase free tier rate limits | accept | Internal tool, low traffic. Rate limits unlikely to be an issue at MVP scale. |
| T-01-SC | Tampering | npm install during build | mitigate | `package-lock.json` (lockfileVersion 3) present — Vercel uses `npm ci` semantics with lockfile. No unvetted packages introduced in this phase (no new installs). |
</threat_model>
