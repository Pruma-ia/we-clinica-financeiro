<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OAuth callback → Supabase | PKCE flow; Supabase validates code_verifier against code_challenge |
| App layer whitelist check | fetchPerfilPermitido queries DB; RLS is additional backstop |
| .env.local on disk | Contains anon key (public-safe) — but gitignored to avoid accidental commit |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | OAuth callback without PKCE | accept | @supabase/supabase-js uses PKCE by default; code_verifier stored in sessionStorage; Supabase validates server-side. |
| T-02-02 | Elevation of Privilege | User bypasses AcessoNegado | mitigate | RLS `is_permitido()` function enforces access at DB layer even if client-side check is bypassed. Dual gate. |
| T-02-03 | Information Disclosure | .env.local committed to git | mitigate | Task 1 explicitly checks .gitignore. .env.local contains only anon key (public-safe), but good hygiene. |
| T-02-04 | Tampering | Session token in localStorage | accept | Supabase JWT is signed; tampering invalidates signature. autoRefreshToken ensures valid tokens stay current. |
| T-02-05 | Denial of Service | Zombie session after AcessoNegado | mitigate | AcessoNegado auto-calls signOut() after 8 seconds, clearing localStorage session. Prevents re-entry without re-auth. |
| T-02-SC | Tampering | npm install during build | mitigate | No new packages installed in this plan. package-lock.json ensures deterministic installs. |
</threat_model>
